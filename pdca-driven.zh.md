# PDCA 驱动的软件开发

## 为什么是 PDCA？

软件开发从来不是线性的。需求会变，理解会深，问题会在实现时暴露。传统的瀑布模型试图在开始编码前规划一切，敏捷开发拥抱了变化但往往缺乏结构化的质量保障。

**PDCA（Plan-Do-Check-Act）** 是一个经典的持续改进循环，源于质量管理领域。PDCA Kit 将这个循环应用到 AI 辅助的软件开发中——不是把它当作一个新概念，而是还原软件开发的本质：**计划、执行、检查、改进**。

## PDCA 循环在软件开发中的映射

```

    │                                              │
    ▼                                              │
  PLAN                  DO                  CHECK
  ──────              ──────               ────────
  说清楚要做什么      按计划实现           检查质量是否达标

  · 宪法原则          · 逐任务执行          · 跨文档一致性分析
  · 功能规格          · TDD 开发            · 代码审查
  · 澄清需求          · 增量构建            · 测试验证
  · 技术方案                                    │
  · 任务拆分                                    │
    │                                           │
    │                                           ▼
    │                                         ACT
    │                                       ────────
    │                                       部署 & 反馈
    │                                       
    │                                       · 部署到目标环境
    │                                       · 发布管理
    │                                       · 收集反馈
    │                                           │
    └─────────────────────── 反馈驱动下一轮 ─────┘
```

### PLAN：把想法变成可执行的方案

这是整个循环的基础。模糊的想法必须被转化为清晰的、可验证的规格。

**核心原则：先说清楚「做什么」和「为什么」，再说「怎么做」。**

很多开发者在有了想法之后立刻开始写代码。PDCA 要求先在 Plan 阶段完成以下步骤：

1. **宪法（Constitution）** — 定义项目的治理原则。什么样的代码算好代码？测试标准是什么？架构约束有哪些？这些原则在后续所有阶段都会被执行和检查。

2. **规格（Define）** — 用自然语言描述功能需求。重点是说清楚**用户需要什么**以及**为什么需要**。不涉及技术选型，不涉及实现细节。遇到不明确的地方，用 `[NEEDS CLARIFICATION]` 标记出来，而不是猜测。

3. **澄清（Clarify）** — 在进入技术方案之前，通过结构化的问答消除规格中的模糊点。这是 Plan 阶段的质量关口——带着不明确的规格进入开发，等于在沙漠中不拿地图就出发。

4. **方案（Plan）** — 将业务需求转化为技术方案。选择技术栈、设计数据模型、定义 API 契约。每一个技术决策都要追溯到具体的需求。

5. **任务拆分（Tasks）** — 将技术方案拆解为可执行的任务列表。明确依赖关系，标记哪些任务可以并行执行，确保每个任务都有明确的产出。

Plan 阶段的产出：**一份被澄清过的功能规格、一份有依据的技术方案、一份可执行的任务列表。**

### DO：按计划执行

Do 阶段不是自由发挥，而是严格执行 Plan 阶段产出的任务列表。

**核心原则：一次一个任务，步步可验证。**

1. **任务驱动** — AI 按照 tasks.md 中定义的任务逐一执行。不会跳过，不会自作主张添加「看起来不错」的功能。

2. **TDD（测试驱动开发）** — 先写测试，确认测试失败，再写实现代码。这不是教条，而是确保每段代码都有对应的验证逻辑。

3. **增量构建** — 每个任务完成后都是一个可验证的增量。不是全部写完再检查，而是做完一个验证一个。

Do 阶段的产出：**符合规格的、经过测试的、可运行的代码。**

### CHECK：质量不是事后检查

Check 不是等到全部写完才做，而是贯穿始终。PDCA 的 Check 阶段有多维度的质量保障：

**核心原则：质量不能「加上去」，只能「建进去」。**

1. **跨文档一致性分析（Analyze）** — 规格、方案、任务三者之间是否一致？有没有规格中的需求在方案里没有被覆盖？有没有方案中的技术决策在任务中没有体现？

2. **代码审查（Review）** — 对照规格和方案审查实现代码。不是笼统地说「代码写得不错」，而是逐条确认：这个功能实现了规格中的第几条需求？这个技术选择符合方案中的架构原则吗？

3. **测试验证（Test）** — 运行测试，确保功能正确。不仅是单元测试，还包括集成测试和端到端测试。

4. **质量清单（Checklist）** — 生成项目定制的质量检查清单，像「英文的单元测试」一样验证需求的完整性和清晰度。

Check 阶段的产出：**一份质量报告，明确指出哪些达标、哪些需要修复。**

### ACT：部署、发布、然后回到 PLAN

Act 阶段将验证过的代码推向生产环境，并将经验反馈到下一轮循环。

**核心原则：每一次部署都是下一次改进的起点。**

1. **部署（Deploy）** — 将功能部署到目标环境。部署失败时有回滚机制。

2. **发布（Release）** — 管理版本、生成变更日志、确认发布就绪。

3. **反馈循环** — 生产环境的表现、用户反馈、发现的问题，都成为下一轮 Plan 阶段的输入。这不是一个线性流程，而是一个不断旋转的飞轮。

Act 阶段的产出：**一个已部署的功能 + 下一轮改进的输入。**

## 为什么 PDCA 循环适合 AI 辅助开发？

AI 编码工具让「从想法到代码」的速度大幅提升，但速度提升也放大了问题：

- **没有 Plan**：直接让 AI 写代码，AI 会猜测你的意图。猜测错误的方向会越走越远。
- **没有 Check**：AI 生成的代码看起来没问题，但可能缺少边界处理、不符合架构原则、或者根本没有测试。
- **没有 Act**：代码写完了但没有部署验证，永远不知道在真实环境中是否正常工作。

PDCA 循环通过结构化的阶段门控，让 AI 在每个阶段都有明确的输入、产出和验证标准。AI 不是被要求「写一个聊天应用」，而是被引导完成一个可追溯的过程：

```
你说 → 规格 → 澄清 → 方案 → 任务 → 实现 → 分析 → 审查 → 测试 → 部署 → 反馈 → 你说（下一轮）
```

每一步都有明确的产出物，每一步都可以回溯。

## PDCA Kit 中的 14 个命令

PDCA Kit 提供了 14 个 AI 命令，覆盖完整循环：

| 阶段 | 命令 | 做什么 |
|------|------|--------|
| **Plan** | `/pdca.constitution` | 建立项目治理原则 |
| **Plan** | `/pdca.define` | 将想法转化为结构化规格 |
| **Plan** | `/pdca.clarify` | 消除规格中的模糊点 |
| **Plan** | `/pdca.plan` | 制定技术实现方案 |
| **Plan** | `/pdca.tasks` | 拆解为可执行任务 |
| **Plan** | `/pdca.taskstoissues` | 任务转 GitHub Issues |
| **Do** | `/pdca.implement` | 按任务列表逐一实现 |
| **Check** | `/pdca.analyze` | 跨文档一致性分析 |
| **Check** | `/pdca.checklist` | 生成质量验证清单 |
| **Check** | `/pdca.review` | 对照规格审查代码 |
| **Check** | `/pdca.test` | 生成并执行测试 |
| **Act** | `/pdca.deploy` | 部署到目标环境 |
| **Act** | `/pdca.release` | 发布管理 |
| **Act** | `/pdca.fallback` | 部署失败时回滚 |

命令之间有 handoff 关系：`constitution` → `define` → `clarify` → `plan` → `tasks` → `implement` → `analyze/test` → `review` → `deploy` → `release` → 回到 `define`。

## 模板驱动：用结构约束 AI 的输出质量

PDCA Kit 的核心不只在命令本身，而在每个命令背后的模板。这些模板不是简单的填空文档——它们是对 AI 行为的结构化约束：

### 防止过早陷入实现细节

规格模板明确区分「做什么」和「怎么做」：
- 规格阶段**禁止**提及技术栈、API 设计、代码结构
- 把 AI 的注意力锁定在用户需求和业务场景上

### 强制标记不确定项

模板要求使用 `[NEEDS CLARIFICATION]` 标记任何不明确的地方，禁止 AI 自行猜测。比如：

```
[NEEDS CLARIFICATION: 用户认证方式未指定 — 邮箱密码？SSO？OAuth？]
```

### 门控机制

方案模板中有「Phase -1 门控」——在进入实现之前，必须通过一系列检查点：
- 项目数量是否 ≤3？（防止过度工程化）
- 是否直接使用框架而非过度封装？
- 契约是否已定义？

通过这些门控意味着 AI 必须先回答「为什么这么设计」再开始实现。

### 清单驱动的自我审查

每个模板都内置了检查清单，让 AI 在产出后自我审查：
- 还有未标记的模糊点吗？
- 每个需求都可测试吗？
- 成功标准可衡量吗？

## 可扩展：适配你的工作方式

PDCA Kit 不是强加一种固定的开发流程。通过扩展（Extensions）和预设（Presets）系统，你可以：

- **扩展**：增加新的命令或开发阶段。比如增加安全审计阶段、性能测试阶段。
- **预设**：覆盖模板内容，适配你的团队术语、组织标准、合规要求。

优先级机制：项目覆盖 > 预设 > 扩展 > 核心默认。你在项目中覆盖一个模板，不会影响其他项目；你安装一个预设，所有项目都会应用。

## 总结

PDCA 驱动的软件开发不是什么新发明。PDCA 循环已经存在了几十年，软件开发本身也一直在「计划 → 构建 → 测试 → 发布」的循环中运转。

PDCA Kit 做的是**将这个循环结构化、可执行化、AI 友好化**。让 AI 编程不再是一句提示词到一坨代码的赌博，而是有规划、有执行、有检查、有反馈的工程实践。

**计划你的工作。执行你的工作。检查你的工作。根据反馈改进。如此循环。**
